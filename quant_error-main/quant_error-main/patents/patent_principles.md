# 软硬结合专利设计原则

本文档用于总结本项目后续设计软硬结合专利 idea 时应遵循的基本原则。这里的“专利 idea”不要求已经完成硬件实测或产品验证，但应当能够通过仿真、公式建模或系统流程模拟来说明其潜在效果。

## 1. 从真实系统瓶颈出发，而不是从算法名称出发

软硬结合专利不应只是将某个算法放到 GPU、NPU 或其他硬件上执行，而应首先明确现有系统中的真实瓶颈。

在本项目语境下，可能的瓶颈包括：

- 低比特量化误差集中在谱空间长尾方向；
- 不同层、不同模块、不同 token 对量化误差的敏感性不同；
- 在线 SVD、秩选择、混合精度调度会引入额外时延；
- 当前 GPU/NPU 的 GEMM 路径难以高效支持“低秩高精度 + 残差低比特”的组合执行；
- 静态量化策略难以根据激活语义、运行时误差信号或时延预算自适应调整。

一个好的软硬结合专利 idea 应当说明：现有硬件执行模型为什么无法自然、高效地解决该瓶颈，以及引入特定软硬协同机制后系统行为如何改变。

## 2. 将算法变量映射为硬件可观测信号

纯算法方案可以笼统地说“根据误差选择量化精度”。软硬结合方案则应进一步说明这些误差、状态或决策依据如何被硬件观测、统计、缓存和使用。

可作为硬件可观测信号的量包括：

- 奇异值能量比例；
- 残差范数；
- 激活 outlier 比例；
- token 语义簇编号；
- 层级或模块级敏感性分数；
- FP4/FP8 量化中的饱和次数、裁剪次数；
- 量化后零值比例；
- 局部重构误差估计；
- GEMM 路径拥塞、执行周期或时延预算。

关键原则是：不要让“智能决策”悬浮在软件策略中，而要将其落到硬件计数器、片上统计单元、控制寄存器、执行描述符或调度器可见的信号上。

## 3. 明确至少一个硬件参与点

软硬结合专利必须有清晰的硬件抓手。该硬件抓手不一定是完整的新芯片，也可以是现有架构中的新增模块、控制通路或数据路径。

常见硬件参与点包括：

- SM/NPU core 内的本地控制器；
- 量化/反量化单元；
- 混合精度 GEMM 调度器；
- tensor core 数据路径；
- 片上 SRAM/shared memory 的布局策略；
- 硬件计数器；
- DMA 或数据搬运引擎；
- kernel fusion 执行单元；
- 精度选择寄存器；
- 稀疏/低秩元数据解码器；
- 编译器生成并由硬件消费的执行描述符。

专利 idea 应当说清楚：该硬件参与点接收什么输入、执行什么判断或操作、输出什么控制结果，以及该结果如何改变后续计算流程。

## 4. 避免只写“在硬件上执行某算法”

仅仅声称“在 GPU 上执行某种谱感知量化算法”通常较弱，容易被理解为软件或算法的常规部署。

更强的表达方式应包括：

- 哪个硬件模块负责观测；
- 观测哪些中间信号；
- 哪个本地控制逻辑负责决策；
- 决策结果写入哪个寄存器、描述符或调度队列；
- 后续硬件数据路径、精度路径或执行路径如何改变；
- 由此如何减少时延、同步、数据搬运或量化误差。

例如，可以将“执行谱感知量化”表述为：在矩阵乘法执行前，由片上统计单元计算激活残差分布，并由本地精度控制器根据残差谱能量选择低秩 head 的秩和 residual 的量化格式，随后将选择结果写入混合精度 GEMM 调度描述符，使 tensor core 对 head 和 residual 采用不同计算路径。

## 5. 将运行时动态性作为重要价值来源

硬件参与的必要性通常来自低延迟、细粒度和运行时动态决策。相比完全离线或静态的软件策略，软硬结合专利应优先强调运行时动态性。

可利用的动态性包括：

- 根据 batch 内 token 激活变化动态选秩；
- 根据层级敏感性动态切换 FP4、FP8、BF16 等精度；
- 根据量化误差计数器动态调整 scale；
- 根据端到端时延预算动态选择 head 保留比例；
- 根据片上资源占用动态融合或拆分计算；
- 根据收敛状态动态提前终止迭代；
- 根据 outlier、残差范数或裁剪率动态选择补偿路径。

动态性越强，越能说明传统静态量化或编译期策略不足，也越容易引出片上控制器、反馈路径和硬件调度机制。

## 6. 用可模拟指标定义专利 idea 的效果

专利 idea 不必已经完成真实硬件验证，但应当能够被模拟、建模或通过实验曲线支撑。

可用于说明效果的指标包括：

- 相比 BF16 基线的理论时延比；
- 相比统一 FP4/FP8 量化的误差下降；
- 下游任务精度保持率；
- 层级或模块级精度损失；
- 片上数据搬运量减少；
- kernel 启动次数减少；
- 混合精度 GEMM 调用次数变化；
- SVD 或近似 SVD 次数减少；
- 低秩 head 占比与精度/时延曲线；
- 饱和值、零值比例、残差范数等统计指标变化；
- 能耗、带宽或 SRAM/HBM 访问次数变化。

每个专利主张最好都能落到一个仿真实验、流程图、公式模型或对比表中。

## 7. 清楚定义软件与硬件之间的接口

软硬结合并不意味着软件消失，而是软件负责配置、边界条件、编译期分析或 runtime 策略，硬件负责低延迟局部执行和动态决策。

可能的软件-硬件接口包括：

- 软件侧生成层级敏感性表；
- 编译器将量化策略编码成执行描述符；
- runtime 将时延预算、精度预算或误差阈值写入控制寄存器；
- 硬件根据描述符执行精度调度；
- 片上控制器将误差信号或状态摘要反馈给 runtime；
- 下一 batch、下一层或下一 token 根据反馈更新策略；
- 编译器生成 fused kernel 或硬件可识别的任务图。

好的软硬结合专利通常是“软件给出策略边界，硬件做低延迟局部决策”，而不是让软件或硬件单独承担全部逻辑。

## 8. 保护点从窄到宽分层设计

一个好的专利 idea 应该具有多层保护范围，避免只保护一个过窄的具体实现。

可以按如下层次设计：

- 最窄层：针对 Metis/SVD/低秩 head + residual FP4 的具体执行方案；
- 中间层：针对谱感知量化中的动态秩选择和混合精度调度；
- 更宽层：针对任意“高敏感子空间高精度、低敏感残差低精度”的硬件执行机制；
- 最宽层：针对基于运行时误差信号的片上自适应精度控制架构。

这样即使最具体方案被认为不够新颖或适用范围有限，也可能保住中间层或更宽层保护范围。

## 9. 将硬件新颖性放在控制路径，而不只是计算路径

矩阵乘法、量化和低比特 tensor core 已经较成熟。软硬结合专利更容易形成差异的地方，往往是控制路径和反馈机制。

重点问题包括：

- 谁判断使用什么精度；
- 什么时候判断；
- 判断依据来自哪里；
- 判断结果如何进入执行单元；
- 是否避免主机同步；
- 是否避免软件重新 launch kernel；
- 是否支持层内、token 内或块内动态调整；
- 是否在片上形成闭环控制。

专利 idea 不一定要发明新的乘法器，但应当尽量发明新的硬件调度、反馈控制或数据路径选择机制。

## 10. 始终回答三个“不可替代性”问题

每个软硬结合专利 idea 都应回答以下三个问题：

1. 为什么不能只靠软件离线完成？
2. 为什么不能只靠现有 GPU/NPU kernel 完成？
3. 为什么加入该硬件机制后，效果或效率会发生本质变化？

如果这三个问题无法回答，idea 容易退化为纯算法专利。

在本项目中，一个较强的回答方向是：谱敏感性和激活分布具有运行时变化，若由 CPU/runtime 处理在线判断，会引入主机-设备同步、kernel launch 和数据搬运开销；因此需要片上统计、片上控制和混合精度执行路径协同，才能在低时延下完成自适应量化和执行调度。

## 11. 面向本项目的优先专利方向

结合当前项目内容，后续可以优先围绕以下三类方向设计专利 idea：

1. 谱感知混合精度硬件执行架构：低秩 head 走高精度路径，residual 走 FP4/FP8 路径。

2. 片上动态秩/精度控制器：根据激活残差、量化裁剪率、层级敏感性和时延预算动态选择 $k_p$、$k_a$ 或量化格式。

3. 面向 Metis 的 fused kernel 或 tensor core 调度机制：将近似 SVD、head/residual 分解、四项 GEMM 和残差合并组织为一个软硬协同执行流。

## 总结

软硬结合专利 idea 的设计原则可以概括为：

> 以算法误差或系统时延瓶颈为起点，把关键决策变量变成硬件可观测信号，通过片上控制、调度或数据路径改变执行方式，并用可模拟的精度、时延、能耗或数据搬运收益来证明硬件参与的必要性。
